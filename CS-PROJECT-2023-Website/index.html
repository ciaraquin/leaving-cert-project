Meeting the Brief






Investigation and Plan


The openness of the brief came with positive and negative aspects. I found choosing a game to model the hardest aspects of this project. While it was simple to propose games to model, whether they could meet the requirements was a different question. I cycled through different ideas (which can be seen to the right) documenting how they would or would not meet the requirements outlined in the brief before settling on one; below, I will talk about some of these.



Uno

My first idea of a game to model was Uno. It appeared to fulfil most of the requirements, but it ended up being more difficult than I anticipated. Uno is a card matching game with a deck of 108 cards, including special cards that would change depending on the situation they were used in, making the code quite long and difficult. I struggled while trying to write a simplified version of the game. I was also not sure how this game would be played for the simulation portion of the project, so I disregarded this idea.



Wordle

After moving on from Uno, I came to Wordle, a game in which the player must guess a 5-character word in 6 moves. After doing a substantial portion of research, I determined that it would not be a viable choice for meeting the brief. The program would have to take an input of five-letter words, validate them, and highlight the letters in the chosen word. While some of this would be simple, I could not find an efficient method to validate the words entered by the user. Although it would have met some of the requirements, such as inputs and data analysis, to make the game fit the requirements, generous portions would have to be changed. This would result in the game being far from the original.



Battleship

Disregarding the ideas above, I decided on modelling battleship. This game is simple in concept, capable of simplification, and can be played in a variety of modes, including single player, multi-player, and simulation play. It can also accept inputs, has data that can be stored and used later for analysis and could have parameters that can change within the game. With my aim being that of having a complete project within the available time in mind, I decided to create a simple base that I could improve upon if time allowed, which will be discussed later in this report.



Objectives

I set the following objectives for the development of the code and to demonstrate my skill in developing a usable and interesting game.

  1. Have functioning versions of the game for all game modes.
  2. Store variables from the game, including hits and misses in a CSV file.
  3. Change guesses available to the player to see how this effects the game.
  4. Show the location of ships if the player loses.
  5. Make difficulty levels.
  6. Ask users to enter their own ship locations in multiplayer mode.
  7. Allow the user to enter how many times the simulation will run.




Design


Single-player mode

Here there will be a set of three pre-selected ship layouts that will be selected at random. The player will then enter a coordinate, and the board grid will change depending on whether the coordinate hits or misses a ship. This will be repeated for the user to guess multiple times. If three ships are hit, the game is won. This could be extended to add more ships in the future. If the player exceeds 20 guesses, the game is over, and the player has lost. At the end of the game, the type of game, number of hits and misses, and result of the game will be recorded in the CSV file for further analysis.


Multiplayer mode

In multiplayer mode, each player will place their own three ships by entering three coordinates. The console will then clear, and the second user will design their layout. The users will then take turns entering coordinates to hit ships. The rules will then follow those of single-player mode, with the board changing depending on if a coordinate was a hit or a miss. The first player to hit three ships wins the game. At the end of the game, the type of game, number of hits and misses, and result of the game will be recorded in the CSV file for further analysis.


Simulation play

The computer will use Python's random function to enter coordinates against the three pre-selected ship layouts. More coordinates will be entered by the computer until either all ships have been sunk and the game is won, or the computer runs out of turns. At the end of the game, the type of game, number of hits and misses, and result of the game will be recorded in the CSV file for further analysis.


Modelling and abstraction

I am using modelling in my project by creating a text-based version of the board and by marking the positions of ships when they have been hit or missed.

I am using abstraction in my project by removing some of the unnecessary aspects of the board game Battleship in order to adapt it to my project and execute it in Python. Some of the details of Battleship I disregarded, including the physical board and physically marking hits and misses, which I replaced with text output. I also removed ships greater than one unit in length to make the game easier to implement.




Implementation


Single-player mode

I started the programming section of this project with single-player, deciding it would be easier to adapt this code for multiplayer and simulation modes. Firstly, I created a nested list for the preselected ship layouts, initialised the variables for hits and misses, and, using a for loop, I created the text-based game board. Through using Python's random library, one of the three pre-set boards is selected randomly, I then add the locations of these ships to a separate list so it can be printed at the end to reveal the locations of the ships. While guesses stay under 20 and the user has not already won, a loop prompts the user to enter a coordinate, then determines if it was a hit or miss, and prints the board again, marking the board with a ‘1’ or an 'X’ accordingly. The while loop then repeats until the user either wins or runs out of guesses. If the user runs out of guesses, the program will reveal the locations.


Multiplayer mode

I found multiplayer mode to be the hardest out of the modes. I tested multiple methods of executing the code, and in the end, I used my flow chart to work out how to program this section. First, I adapted the single-player version of the game. I made two separate nested lists to allow the user to enter the location of their own ships and made four variables to track each player's hits and misses. By importing the time library, I was able to clear the console after a certain amount of time to hide the locations of the other players ships. Beyond this point, the game stays broadly similar to how single-player works. The user enters a coordinate, and the program determines if it is a hit or a miss and places a ‘1’ or an ‘X’ accordingly. The main change I had to implement for this version of the game was that the console now clears after every turn to make the game easier to follow for both players and, to hide the progress of the other player. If one of the players wins, or the player runs out of guesses, the game ends, and the code displays the winner. Then a summary is displayed of hits and misses, and appends data to the CSV file.


Simulation play

I found coding the simulation mode to be the easiest out of all three modes. First, I changed the code so that a random item would be picked out of the list ‘row’ to get the letter part of the coordinate and then use randint to get the other part of the coordinate. I used the code from single-player and altered the while loop so that the game would repeat the number of times the users had entered. Then I used the code from single-player to mark hits and misses. After the game is repeated the number of times the user specified, a summary is shown of the number of games won or lost.


Problem

Many of the complications I encountered in programming the model seemed difficult at first but ended up being small syntax errors and were easy to fix when I found them.

One major factor of game play that I simply overlooked was the possibility of guessing the same coordinate more than once. Testing the single-player version of the game was the first time I came across the issue, although it ran through all modes of play. After finding the problem, I found that if a player had found one ship, they could simply guess the same coordinate two more times, and the program would determine that the player had won the game.

Searching through the code, I found out where the problem was. It was in the if statement on line 95. I soon realised the issue was in the order of the if statements. The error was that the first if statement checks whether the coordinate was a hit or miss. Although I did have code determine if coordinate had been hit previously, the case would never run as it was located below this first if statement.

To fix this problem, I had to change the if statement so that the first case would check whether the coordinate had been hit previously and that the second case would check for a hit and then for a miss. I then had to replace all game modes with the fixed code so that all game modes would function properly.





Algorithm

The code shown below was probably the most important for my project as it is central to the operation of the game. First, the code checks whether the coordinate has already been hit or missed by checking if the coordinate on the board has already been replaced by an ‘X’ or an '1'. If so, the code prints ‘You've already guessed this coordinate!’ and makes no changes to hits or misses. The elif statement on line 99 checks whether the coordinate the user entered matches one that is in the selected list. If the coordinate was a ‘hit’, it marks a ‘1’ on the board, adds one to the hit’s variable, and subtracts one from the guesses left. After that, there is another else statement. Within this else statement, there is an if statement that will not run, as I went back and put boundaries on the coordinates that could be entered. The else statement determines that the coordinate entered was a miss, marks an ‘X’ on the board, adds one to the misses variable, and subtracts one from the guesses left.





Testing


Throughout my project, I used both unit and system testing to determine whether my project was working as I had intended.


Unit testing

One of the main sections of code I used unit testing for was the multiplayer section.

I made sure to do unit testing as, through my Leaving Cert Computer Science course, I have learned that if proper unit testing is done in early development when needed can save a lot of time in the long run. Timing being an essential aspect of this project, I made sure to use unit testing when appropriate.

I put the multiplayer function in a separate Python file and worked through the code by itself without interference from the rest of the code, and I was able to work through my errors this way.

In doing this, I was also able to identify dependencies on other code in the project.


System testing

The main types of system testing I focused on in this project were usability testing and regression testing. When making the artifact for this project, I would consistently test the user’s ease of use of the application, the ability of the game to meet the project requirements, and to make sure none of the changes made over the course of the development process had caused new bugs.

Evaluation


Objectives complete

  • Through sticking to a timeline, I was able to allocate time to the most important aspects of the project.
  • By the end I had completed most of my objectives; I was able to make a basic version of the game battleship, store hits and misses in a CSV file, use this data to for analysis, reveal the locations of the ships after the game had been completed, change parameters by changing the number of guesses allowed and allowed the user to enter how many times the simulation play will run.
  • One objective that I didn’t have time to complete was difficulty levels.

How I would improve my project in the future

  • To develop the project further I would add the objective I did not have time to complete. I think adding game difficulty levels would be a great addition to the game.
  • For the easy mode I would change the guesses variable to 25 but keep the same number of ships, for medium I would keep the guesses variable at 20 but increase the number of ships to 5. For hard mode I would decrease the guesses variable to 15 and keep the number of ships at 5.

Hypothesis and changing parameters
  • Using the model I created, I was able to answer my hypothesis of ‘which pre-set board has the highest chances of winning when total_guesses = 20. Figure 1 shows that, by very small margins that pre-set layout 2 has more wins than the others.
  • For changing the parameters, I decided to change the number of total_guesses available to the computer.
  • Using the model, I was able to graph the different ratios of wins to losses based on both the pre-set layout and the number of guesses available.
  • Interestingly, by doing so I was able to recognise a pattern for the relationship between the guesses available and the percentage wins. When variable guesses is increased by 5, win percentage is increased by around 10%.
